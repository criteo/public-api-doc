<Note>
  **Getting Started**

  The endpoints here will **only support the booking and approval process for demand-side users**. The GET call will allow demand-side users to see the current status of their creative proposal. 

  In the booking approval v1, we introduce two endpoints. A GET *(Fetch proposal)* and a POST *(Submit Proposal)* for review processes.

  **Brands**

  Brands or demand accounts, when calling the POST *(Submit Proposal)* will trigger the creative Proposal for the retailer to review. Creative booking status will remain pending, and the line-item will not begin to serve until the retailer reviews and approves the creative.

  **Retailers**

  Retailers submitting direct sold line items are not required to review and approve their creatives. So if you have an app with API access to a retailer or a supply account, calling the *submit proposal* endpoint will automatically approve the creative proposal. Following the creative approval step, the line item will activate as long as the line item is correctly set up.

  **⚠️ Important note for retailers**

  Applications using this endpoint must have the proper workflow setup to ensure only users with the appropriate permissions within their application can trigger this endpoint. In the case of this endpoint, we highly recommend that only retail managers who have the approval to review and approve creatives can call the POST endpoint. Failing to set up the correct workflow within your application can risk having the proposal approved before the retail managers can review and approve the creative proposals.

  **Consult with your application UI and UX teams to ensure you have the correct user workflow for these endpoints.**
</Note>

## **Attributes**

<ParamField path="id" type="string" required>
  The external line item id used to retrieve and submit proposal

  **Values:** -
</ParamField>

<ParamField path="negotiationState" type="enum">
  The state of the proposal.

  **Values:** `Invalid`, `Modified`, `Submitted`, `Reviewed`, `Approved`
</ParamField>

<ParamField path="bookingStatus" type="string">
  The status of the proposal for the purpose of booking.

  **Values:** `NotSubmitted`, `Submitted`, `Rejected`, `Approved`
</ParamField>

<ParamField path="runnableStatus" type="string">
  The status of the proposal for the purpose of running in delivery.

  **Values:** `NotSubmitted`, `Submitted`, `Rejected`, `Approved`
</ParamField>

<ParamField path="comment" type="string">
  Comment supplied by supply side. Required only if some part of the proposal is rejected. An automatic comment will be provided for auto-approved proposals.

  **Values:** `NotSubmitted`, `Submitted`, `Rejected`, `Approved`
</ParamField>

<ParamField path="createdAt" type="string">
  Creative proposal creation date

  **Values:** date-time
</ParamField>

<ParamField path="updatedAt" type="string">
  Date of last update made to the creative proposal

  **Values:** date-time
</ParamField>

## **Get Line Item Creative Proposal**

<EndpointBadge method="get">
```http
https://api.criteo.com/{version}/retail-media/preferred-deal-line-items/{lineItemId}/proposal
```
</EndpointBadge>

**Sample Request**

```bash
curl -L -X GET 'https://api.criteo.com/{version}/retail-media/preferred-deal-line-items/408713406654238720/proposal' \
-H 'Accept: text/plain' \
-H 'Authorization: Bearer <MY_ACCESS_TOKEN>'
```

**Sample Response**

```json
{
    "data": {
        "id": "408713406654238720",
        "type": "RetailMediaProposalStatusModel",
        "attributes": {
            "negotiationState": "Approved",
            "bookingStatus": "Approved",
            "runnableStatus": "Approved",

            "createdAt": "2023-02-01T20:01:09.7427245",
            "updatedAt": "2023-02-01T20:03:47.1133981"
        }
    }
}
```
