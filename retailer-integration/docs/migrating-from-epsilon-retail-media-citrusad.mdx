# Migration Summary

* Same flow end-to-end: **request ads → show them → track → report sales**.
* Main shift: **placement-based requests (CitrusAd)** → **page/context-based requests (Criteo)**.
* Catalog work: **remap product IDs + required fields** to Criteo’s feed rules.
* Rendering: **render by placement returned in the response**, not by what you requested.
* Measurement: **tracking endpoints (CitrusAd)** → **beacons in the response (Criteo)**.
* Sales: **send a `trackTransaction` on confirmation** with **all order items**.
* Reuse: **your pages/placements and taxonomy mostly carry over**; update the wiring + mappings.
* Retailer effort: **integration plumbing**; platform handles **ad selection + attribution**.

## Catalog

**What CitrusAd does today**

CitrusAd expects you to create/sync a catalog and products where each product is identified by `gtin`, enriched with `filters`, and includes `tags` such as `imageurl:` and `name:`. Inventory can be provided (or omitted, defaulting to 1), and products are synced into a specific `catalogId`.

**What Criteo's Retail Media expects**

Criteo expects a product feed with mandatory fields (e.g., `id`, `title`, `description`, `link`, `image_link`, `price`, `availability`, and others) that **must match the identifiers you will later send in API requests**. Product IDs must be stable, unique, and consistent to preserve attribution and reporting.

**Minimum viable implementation (retailer work)**

* Choose the **single product identifier** you will use as Criteo `id` and ensure it is stable over time.
* Map CitrusAd product data into Criteo mandatory feed fields (at minimum: `id`, `title`, `description`, `link`, `image_link`, `price`, `availability`, and required product classification fields).
* Ensure your onsite pages can reference the same SKU IDs you send in the feed (so request parameters and transaction reporting match ingestion).
* Populate product classification consistently (your category taxonomy → Criteo required product type fields).
* Validate that availability/pricing fields match what shoppers see (avoid mismatch between feed and onsite).
* If you support seller/marketplace, ensure product IDs and seller IDs are distinct and consistent.

**Validation checklist**

* [ ] Every product has a stable, unique identifier (Criteo `id`) and it matches what the site can send in requests.
* [ ] Mandatory Criteo feed fields are present and non-empty (ingestion-blocking fields are populated).
* [ ] Your taxonomy/category mapping is consistent across the whole feed (no mixed conventions).
* [ ] PDP URLs and image URLs resolve correctly for a representative product set.
* [ ] Availability and price values match what is shown to shoppers.
* [ ] The same SKU IDs appear consistently across feed, ad responses, and sales tracking payloads.

**Common pitfalls**

* Using a different identifier in the feed than what your frontend/backend can send later (leads to missing matches and broken attribution).
* Changing product IDs over time (explicitly called out as breaking attribution/reporting in Criteo docs).
* Keeping CitrusAd-style `tags` without translating them into required Criteo fields (missing `title`, `image_link`, etc.).
* Inconsistent category strings when mapping to required product classification fields (campaign setup and filtering become unreliable).

## Ad Request

**What CitrusAd does today**

CitrusAd ad generation requests are made to `/v1/ads/generate` and require `placement`, `catalogId`, and `maxNumberOfAds`, plus session attribution fields like `sessionId` (and `customerId` when available). Context is passed per placement (e.g., `searchTerm` for search, `productFilters` for category).

**What Criteo's Retail Media expects**

Criteo ad requests are GET calls to `/delivery/retailmedia` with global parameters (e.g., `criteo-partner-id`, `retailer-visitor-id`, `page-id`) and page/event-specific parameters. The request identifies a **page type via `page-id`** and returns placements configured for that page (placement selection is not requested the same way as CitrusAd’s `placement`).

**When something is not explicitly documented in the provided guides (examples):**

* “Search page parameters in Criteo” or “Category page parameters in Criteo” are **not explicitly documented in the provided excerpts above**.
  * **Inference-based approach:** locate the relevant Criteo page-type documentation (e.g., “Search results page”, “Category page”) and build a mapping of required query parameters to your existing CitrusAd `searchTerm` / `productFilters`.
  * **Assumptions:** Criteo provides page-type docs for key onsite pages; these define required query parameters for each page context.
  * **What would confirm/deny:** the presence (or absence) of those page-type docs and parameter lists in the Criteo guide (e.g., headings like “Search Results Page”, “Category Page”, with a “Parameters” section).

**Minimum viable implementation (retailer work)**

* Implement Criteo request construction for `/delivery/retailmedia` including required global parameters.
* Generate and persist a `retailer-visitor-id` and pass it on every request.
* Pass `customer-id` only when the user is authenticated (leave empty otherwise).
* Use the correct `page-id` values for each page type you integrate (as provided by Criteo).
* For each page type, populate the required page-specific parameters (per Criteo page documentation).
* Decide and standardize on a `verbosity` level and ensure your renderer supports it.
* Update your ad-call abstraction to handle multiple placements in one response.

**Validation checklist**

* [ ] Requests include `criteo-partner-id`, `retailer-visitor-id`, and `page-id` as required.
* [ ] Logged-in sessions include `customer-id`; logged-out sessions do not.
* [ ] Each page type sends the correct page-specific parameters (per Criteo docs).
* [ ] Response handling supports multiple placements returned for a single page request.
* [ ] You can switch `verbosity` safely without breaking rendering/tracking.
* [ ] SKU identifiers used in requests/transactions match the feed’s product IDs.

**Common pitfalls**

* Treating Criteo calls like CitrusAd calls (one “placement” per request) and ignoring placement keys returned in the response (ads “disappear” because nothing renders).
* Missing or non-persistent `retailer-visitor-id` (degraded attribution; inconsistent behavior across sessions).
* Sending `customer-id` incorrectly (e.g., always empty even when logged in).
* Using the wrong `page-id` for a page type (placements not returned, or wrong formats returned).
* Not aligning SKU IDs across feed ↔ requests ↔ transactions (breaks matching and reporting).

## Ad Response / Rendering

**What CitrusAd does today**

CitrusAd returns an `ads` array from `/v1/ads/generate`, where each ad includes fields such as `id`, `gtin`, `position`, and optional objects like `discount` and `metadata`. The retailer renders the sponsored products/creatives into the requested placement and uses returned identifiers for downstream tracking.

**What Criteo's Retail Media expects**

Criteo returns a JSON response containing `placements`, each containing formats (e.g., sponsored products) and associated products, plus beacon URLs used for tracking. Retailers must render by placement and format, and wire the returned beacons as required for tracking and measurement.

**Verbosity (Criteo) — what changes and trade-offs**

* **Minimum verbosity** (documented by Criteo) contains core product identifiers and beacons.
* **Full verbosity** (documented by Criteo) provides additional product fields.
* **Retailer impact:** choose the lowest verbosity that still provides the data you need at render time; otherwise, you must source missing merchandising data from your own catalog/PDP services.
* **Trade-off:** full verbosity can simplify rendering (more data in response) but may increase payload size.

**Example** _(illustrative shape only)_:

```json
{
  "status": "OK",
  "placements": [
    {
      "<placement-key>": [
        {
          "format": "sponsored_products",
          "products": [
            { "ProductId": "14567", "ParentSKU": "14567P", "OnLoadBeacon": "<url>", "OnViewBeacon": "<url>" }
          ]
        }
      ]
    }
  ]
}
```

**Minimum viable implementation (retailer work)**

* Parse the Criteo response by `placements` and render each placement key to the correct UI slot.
* For sponsored products, map returned product IDs to your product cards (title/image/price may come from your own catalog if not present).
* Implement beacon firing tied to lifecycle events (render/load vs view vs click), using only URLs returned by Criteo.
* Choose a `verbosity` level and ensure your renderer either uses returned fields or fetches missing fields from your own systems.
* For any enabled display format, follow the format-specific rendering requirements (assets, click behavior, etc.).
* Ensure you can handle “no fill” cases gracefully (empty placements or suppressed products).

**Validation checklist**

* [ ] Response parsing correctly enumerates all placements and formats returned.
* [ ] Each placement key maps to a real UI location (nothing is “dropped”).
* [ ] Product IDs in the response can be resolved to real products in your catalog.
* [ ] Minimum verbosity rendering works end-to-end (or full verbosity is explicitly chosen).
* [ ] Beacons are fired at the correct lifecycle moments and only from returned URLs.
* [ ] Empty/no-ad responses render gracefully without layout breaks.

**Common pitfalls**

* Treating the response as a flat list (CitrusAd-style) and ignoring placement keys (ads never show).
* Depending on response fields that aren’t guaranteed at your chosen verbosity (missing title/image/price at render time).
* Rendering products but not wiring beacons (impressions/views not counted; reporting gaps).
* Mis-mapping placement keys to UI slots (ads appear in the wrong place or not at all).
* Assuming the platform will provide all merchandising fields (often you must use your own catalog at render time).

## Tracking

**What CitrusAd does today**

CitrusAd tracking is reported by the retailer calling CitrusAd “resource” endpoints: impressions go to `/v1/resource/first-i/AD_ID` and clicks go to `/v1/resource/second-c/AD_ID`, using the `adId` returned in the ad response. Calls must be executed within the app or website (backend-to-backend style is emphasized for environments like mobile apps).

**What Criteo's Retail Media expects**

Criteo tracking is driven by beacon URLs included in the ad response, and the docs describe both legacy and standard/universal beacon approaches. Retailers must fire the appropriate beacons (e.g., load/view/click-related) as specified, using the URLs provided in responses.

**Minimum viable implementation (retailer work)**

* For CitrusAd parity: map your existing “impression” and “click” triggers to Criteo’s beacon-based lifecycle triggers.
* Store beacon URLs returned in the response alongside the rendered placement/product so you can fire them later.
* Fire the “load/render” beacon when the ad is rendered, and fire the “view” beacon when viewability conditions are met (as you define per your implementation and Criteo guidance).
* Fire click-related tracking using the mechanism described in Criteo’s beacon docs (using returned URLs, not constructed ones).
* Ensure tracking fires on both desktop and mobile implementations consistently.
* If you previously did backend-to-backend tracking for mobile limitations, ensure your architecture can still call beacon URLs reliably.

**Validation checklist**

* [ ] Every rendered placement/product has its corresponding beacon URLs stored and available at runtime.
* [ ] Load/render beacons fire exactly once per render event (no double-counting).
* [ ] View beacons fire only when viewability criteria are met (and are not missed).
* [ ] Click tracking fires on all click entry points (product tile, CTA, brand zone, etc., depending on format).
* [ ] Removing tracking (intentionally) causes measurable drops (sanity test that tracking is actually connected).
* [ ] Tracking works across device types and page templates consistently.

**Common pitfalls**

* Continuing to call CitrusAd-style fixed endpoints instead of firing Criteo-provided beacon URLs (nothing gets counted).
* Firing beacons too early (on response receipt instead of render) or too late (after navigation), causing under/over-counting.
* Losing beacon URLs when the page re-renders (SPA behavior) and missing events.
* Mixing legacy vs universal beacon logic (unexpected request volume or missing optimization signals).
* Not correlating which placement/product a beacon belonged to (hard to debug missing reporting).

## Sales Tracking

**What CitrusAd does today**

CitrusAd sales tracking is reported by posting orders to `/v1/orders`, where each order can include `customerId`, `sessionId`, `id` (order ID), `orderItems` (with fields like `gtin`, `quantity`, pricing/discount fields), and ad linkage fields such as `adIds` and per-item `adId`. This ties purchases back to ads for attribution.

**What Criteo's Retail Media expects**

Criteo sales tracking is sent as an order confirmation event using `event-type=trackTransaction` with a dedicated `page-id` for that event type, and it requires `transaction-id`, `item`, `price`, and `quantity`. Criteo explicitly requires including **all items in the order, regardless of whether they are sponsored or not**.

**Minimum viable implementation (retailer work)**

* Implement the order confirmation page transaction call using `event-type=trackTransaction` and the correct `page-id` format for your region/environment.
* Always include the complete order contents in `item`, `price`, and `quantity` lists (not just sponsored items).
* Ensure `item` values match the product identifier used in your Criteo feed.
* Send `transaction-id` as your unique order ID.
* If you support promotions/discounts, calculate and report `price` as unit prices consistent with Criteo’s discount guidance.
* Add `currency` when you support multiple currencies; otherwise rely on default behavior.

**Validation checklist**

* [ ] `event-type=trackTransaction` is sent only on the order confirmation page (post-purchase).
* [ ] `transaction-id` matches your internal order ID and is unique per purchase.
* [ ] `item`, `price`, and `quantity` lists have the same length and aligned ordering.
* [ ] `item` identifiers match the feed’s product IDs exactly.
* [ ] All order items are included (sponsored + organic).
* [ ] Discounted orders reconcile: sum(price × quantity) matches expected totals based on Criteo’s unit price guidance.
* [ ] Multi-currency setups include `currency`; single-currency setups behave as expected without it.

**Common pitfalls**

* Reporting only sponsored items (explicitly called out as incorrect for Criteo; causes revenue under-reporting and attribution gaps).
* Misaligned pipe-separated lists (items/prices/quantities shifted → wrong revenue attribution per SKU).
* Sending total line price instead of unit price (Criteo multiplies `price` by `quantity`).
* Using product IDs in transactions that don’t match the feed (transactions can’t be reconciled to products).
